# Project Scope

## Project Name

Telco CDR Lakehouse Modernization

## Objective

The objective of this project is to simulate a modern lakehouse-based telecom CDR processing platform using Databricks.

The project demonstrates how traditional file-based telecom CDR processing can be redesigned with a Bronze-Silver-Gold architecture, data quality checks, quarantine handling, reconciliation, and SQL analytics.

## Business Context

Telecom operators process large volumes of Call Detail Records generated by mediation systems, network switches, roaming partners, and data platforms.

These records are critical for:

- Usage reporting
- Customer analytics
- Billing support
- Revenue assurance
- Reconciliation
- Operational monitoring

Traditional architectures often rely on legacy ETL tools and relational databases. This project shows how the same business flow can be represented using a modern lakehouse pattern.

## In Scope

The project includes:

- Simulated CDR batch generation
- Multiple source system simulation
- Batch-level metadata tracking
- Bronze raw data layer
- Silver validated data layer
- Quarantine layer for invalid records
- Data quality rule implementation
- Duplicate detection
- Gold aggregate tables
- Batch-level reconciliation
- SQL analytics queries
- Dashboard-ready outputs
- GitHub-ready documentation

## Out of Scope

The following items are not included in the first version:

- Real cloud object storage integration
- Real-time streaming ingestion
- Production IAM and security setup
- CI/CD deployment
- Large-scale performance benchmarking
- Production-grade scheduling
- Billing system integration

## Free Edition Scope

This project is implemented on Databricks Free Edition.

Due to Free Edition limitations, the source file landing zone is simulated through generated CDR batches and managed Databricks tables.

In a production-grade implementation, the source files would be ingested from a cloud object storage landing zone such as GCS, S3, or ADLS.

## Expected Outputs

The project will produce the following tables:

### Metadata

- metadata_ingestion_log

### Bronze

- bronze_cdr_raw

### Silver

- silver_cdr_clean
- silver_cdr_quarantine

### Gold

- gold_cdr_usage_hourly
- gold_subscriber_usage_daily
- gold_reconciliation_daily

## Success Criteria

The project is considered successful if:

- Simulated CDR batches are generated
- Raw records are stored in the Bronze layer
- Valid records are stored in the Silver clean layer
- Invalid records are isolated in the quarantine layer
- Gold aggregate tables are created
- Batch-level reconciliation is calculated
- SQL analytics queries return meaningful outputs
- At least one dashboard or dashboard-ready query set is created
